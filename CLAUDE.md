# CLAUDE.md (Project Root)

저장소 전체에 적용되는 **항구적 지침**이다. 프로젝트 성격, 디렉터리 구조, spec과의 관계, 비밀·개인 데이터 관리, 작업 흐름 등 언어·모듈과 무관한 토대를 다룬다.

언어·모듈에 특화된 규칙(백엔드 Java/Spring의 계층·파일 I/O 경계·테스트)은 해당 하위 디렉터리의 `CLAUDE.md`에 있다. 그 디렉터리에서 작업할 때는 루트 + 해당 모듈 `CLAUDE.md`를 함께 따른다.

---

## 1. 프로젝트 성격

**모카(Moka)** — Slack에 자연어로 던진 커피 감상을 구조화해 로컬 JSON으로 기록하고 정적 HTML 노트로 렌더링하는 1인용 개인 에이전트. 개인 실사용이 1차 목적이고, 실사용하며 델타로 개선한 기록 자체가 포트폴리오 자산이다. 일반 사용자 릴리즈는 미래 옵션(spec 비범위)이며, 이 문서는 그와 무관하게 유지되는 토대만 규정한다.

## 2. 저장소 구조

- `specs/` — 기능별 명세(SDD). **모든 기능 요구사항·설계 결정의 source of truth.** 현재 유일 기능: `specs/coffee-note-agent/`.
- 백엔드(Java/Spring) — 소스 트리. 해당 디렉터리의 `CLAUDE.md`가 구현 규칙을 규정한다.
- `data/` — **런타임 산출(원본 데이터)**: 노트 JSON·사진·pending. 커밋하지 않는다(§5).
- `site/` — **런타임 산출(파생물)**: 정적 HTML·썸네일. JSON에서 전체 재생성 가능하므로 커밋하지 않는다.
- (이후 모듈이 추가되면 각 모듈 루트에 `CLAUDE.md`를 둔다.)

## 3. spec과의 관계 (SDD)

- 각 기능은 `specs/<feature>/`에 spec(요구사항)·plan(설계·ADR)·data-model·tasks를 가진다.
- **기능을 구현·수정하기 전에 반드시 해당 spec을 정독한다.** 충돌 시 spec을 우선한다.
- 구현 순서: spec 정독 → tasks의 Phase 순 → 각 작업이 어느 FR/AC/ADR을 충족하는지 의식하며 구현.
- **spec에 없는 동작을 임의로 추가하지 않는다.** 필요하다고 판단되면 spec에 먼저 반영(또는 사람에게 제안)한 뒤 구현한다.
- 설계 결정을 바꿔야 하면 코드만 바꾸지 말고 해당 ADR/spec도 갱신한다. 코드와 문서가 어긋나지 않게 유지한다. 기존 기능 수정은 델타(`changes/<NNNN>-<slug>/`)로 남긴다.

## 4. 공통 원칙

- **right-sizing**: 미래 대비 선제 도입을 하지 않는다. 새 인프라(DB, 캐시, 메시지 브로커, 웹서버 서빙 등)나 무거운 추상화는 *문제가 실제로 관측됐을 때* 도입한다. 필요해 보이면 코드로 들어가기 전에 사람에게 묻는다. 미래 옵션(호스팅형, jpackage 배포 등)을 위한 선제 일반화도 금지 — spec이 요구한 인터페이스 경계(NFR-4)까지만.
- **실사용 우선순위**: 기록이 실제로 쌓이는 것이 최우선 가치다. 화려함보다 "메시지 한 줄 → 저장까지"의 마찰 최소화. 개선은 실사용에서 관측된 불편에서 출발한다.
- **문서-코드 일치**: 결정과 구현이 갈라지지 않게, 둘 중 하나를 바꾸면 다른 하나도 맞춘다.

## 5. 비밀 / 개인 데이터

- 비밀: Slack bot token·app token, OpenAI API 키. 코드·설정·커밋에 하드코딩 금지. 프로젝트 루트 `.env` → 환경변수로 주입.
- **`.env`는 절대 커밋 금지.** `.gitignore`에 포함. 형식 공유는 `.env.example`(빈 값)로.
- OpenAI 키는 종량제 API 키이며, 일반 구독과 별개임을 인지한다.
- **개인 데이터도 비밀에 준해 다룬다.** `data/`(노트·사진)에는 개인 일상 정보가 담기므로 커밋·공개 금지, `.gitignore`에 포함. 레포를 포트폴리오로 공개할 때는 실데이터가 아니라 별도 샘플 데이터로 시연한다.

## 6. 작업 전 체크 (공통)

- 건드리는 기능의 `specs/<feature>/` 문서를 읽었는가.
- 새 동작이 spec 범위 안인가. (범위 밖이면 spec 반영/제안 먼저)
- 추가하려는 인프라/의존성이 §4 right-sizing에 맞는가. (애매하면 묻는다)
- 비밀·개인 데이터(`.env`, `data/`)가 코드/커밋에 들어가지 않는가.
- 작업 디렉터리에 모듈 `CLAUDE.md`가 있으면 그 규칙도 함께 따랐는가.
