# REVIEW.md

코드 리뷰 시 적용하는 기준이다. 모카는 1인용 로컬 에이전트다(루트 `CLAUDE.md` §1, §4 right-sizing). 리뷰의 방향도 같다: **작게 유지하고, 실제로 관측된 필요만 반영한다.** 이 문서와 spec이 충돌하면 spec이 우선한다.

---

## 1. 의존 대상을 이름에 드러낼 것

구현 클래스는 **무엇에 의존하는지 이름만 보고 알 수 있어야 한다.**

- 외부 서비스·기술에 의존하는 구현체는 그 대상을 접두어로 명명한다.
  - 예: `OpenAiAgentClient`, `JsonFileNoteRepository`, `ThymeleafNoteRenderer`, `PlaywrightCardImageRenderer`, `LocalPhotoStore`
  - 나쁜 예: `AgentClientImpl`, `DefaultNoteRepository`, `NoteRepositoryImpl` — `Impl`/`Default` 접두·접미어는 의존 정보를 숨기므로 금지.
- 클래스 내부에서도 외부 의존(HTTP 클라이언트, 파일 시스템, 프로세스 실행 등)은 필드·생성자 시그니처에 드러나야 한다. 메서드 깊숙한 곳에서 몰래 `new`로 외부 자원을 만들지 않는다.

**리뷰 질문**: 이 클래스 이름만 보고 뭘 쓰는지 알 수 있는가? 숨은 외부 의존이 있는가?

## 2. 추상화는 교체 가능성이 실재하는 경계에만

인터페이스는 **"실제로 교체하거나 테스트에서 격리할 이유가 있는 경계"**에만 둔다. 현재 허용된 경계:

| 경계 | 인터페이스 | 이유 |
|---|---|---|
| LLM 호출 | `AgentClient`, `VisionClient`, `AliasGenerator` | 모델/프로바이더 교체 가능성, 테스트 격리 |
| 저장 | `NoteRepository`, `PendingStore`, `PhotoBufferStore`, `PhotoStore` | 저장 방식 교체 가능성(NFR-4), 테스트 격리 |
| 메시지 송수신 (Slack) | `SlackResponder`, `PhotoDownloader`, `ConversationRouter` | 외부 서비스 경계, 테스트 격리 |
| 렌더링 | `NoteRenderer`, `CardImageRenderer` | 템플릿/렌더링 엔진 경계 |

이 표 밖의 새 인터페이스는 **기본적으로 반려**한다. 정말 필요하면 이 표에 행을 추가하는 변경(사유 포함)과 함께 온다.

- 구현체가 하나뿐이고 앞으로도 하나일 클래스에 인터페이스를 씌우지 않는다.
- "나중에 필요할지도 모르는" 제네릭 파라미터, 전략 패턴, 팩토리, 이벤트 버스를 선제 도입하지 않는다.
- 단순 위임만 하는 래퍼 클래스(한 줄 통과 계층)를 만들지 않는다.

**리뷰 질문**: 이 인터페이스를 지우고 구현 클래스를 직접 쓰면 뭐가 나빠지는가? 답이 "없다"면 지운다.

## 3. 데드 코드는 발견 즉시 제거

- 호출자 없는 public 메서드, 사용처 없는 클래스·필드·상수, 주석 처리된 코드 블록, 도달 불가 분기는 **리뷰에서 발견되면 그 PR에서 지운다.** "나중에 쓸지도 몰라서"는 사유가 아니다 — git 히스토리가 보관한다.
- 미래를 위한 파라미터(항상 같은 값이 넘어오는 인자, 항상 null인 옵션)도 데드 코드다.
- spec에서 제거된 기능의 잔재 코드가 남아 있으면 함께 지운다.

**리뷰 질문**: 이 심볼의 사용처를 IDE로 찾았을 때 프로덕션 호출자가 있는가?

## 4. 아키텍처는 이 규모에 맞게

- **헥사고날·클린 아키텍처식 계층 분리를 도입하지 않는다.** port/adapter, usecase/interactor, application/infrastructure 같은 계층 패키지·명명을 만들지 않는다. §2의 인터페이스 경계면 충분하다.
- **일반 도메인 모델링(엔티티-VO-애그리게잇-도메인서비스 구분, 리치 도메인 모델)을 하지 않는다.** `domain/` 패키지는 노트 JSON 구조를 그대로 반영하는 record 중심의 데이터 홀더로 유지한다. 로직은 그 데이터를 쓰는 쪽(agent, render, slack)에 둔다.
- 매퍼 계층·DTO 변환 계층을 습관적으로 추가하지 않는다. 같은 데이터를 표현하는 클래스가 두 벌 생기면 그 자체가 리뷰 지적 대상이다(뷰 전용 가공이 실제로 필요한 `NoteView` 같은 경우만 예외).
- 새 인프라(DB, 캐시, 큐, 웹서버)는 코드 리뷰가 아니라 spec 논의 단계의 문제다. 코드에 먼저 나타나면 반려.

**리뷰 질문**: 이 구조가 "1인용 로컬 앱"에서 실제로 관측된 문제를 푸는가, 아니면 관성적인 패턴 적용인가?

## 5. 패키지는 응집 단위로 작게

- 한 패키지가 **대략 10개 파일을 넘으면 분리 신호**로 본다. 숫자 자체보다 "서로 다른 관심사가 섞여 있는가"가 기준이다.
  - 좋은 선례: `slack/` → `slack/inbound/`, `slack/outbound/` 분리, `agent/` → `agent/prompt/`, `agent/tool/`, `agent/conversation/` 분리.
- 분리할 때는 계층(예: `service/`, `impl/`)이 아니라 **기능·관심사**로 자른다.
- 반대로, 파일 1~2개짜리 패키지를 예비로 미리 만들지도 않는다. 분리는 커졌을 때 한다.

**리뷰 질문**: 이 패키지의 파일들이 한 문장으로 설명되는 하나의 관심사인가?

## 6. 그 외 일반 원칙

위 항목에 걸리지 않는 부분은 일반적인 객체지향 설계 원칙을 따른다. 이 프로젝트에서 특히 자주 보는 것:

- **단일 책임**: 클래스가 "~하고 ~도 한다"로 설명되면 분리를 검토한다. 단, §2·§4에 어긋나는 분리(추상화 계층 추가)로 풀지 말고 클래스 분할로 푼다.
- **불변 우선**: 데이터 홀더는 record, 컬렉션은 방어적 복사 또는 불변 컬렉션.
- **생성자 주입**: 의존은 생성자로 받는다. 필드 주입·세터 주입 금지.
- **실패는 숨기지 않는다**: 예외를 삼키고 빈 값을 돌려주는 코드 금지. 복구할 수 없으면 전파하고, 사용자에게 알려야 하면 spec의 오류 처리 흐름을 따른다.
- **테스트 용이성은 §2 경계로 확보한다**: 테스트를 위해 새 인터페이스를 만들지 말고, 이미 있는 경계(저장·LLM·Slack)에서 격리한다.

## 리뷰 체크리스트 (요약)

- [ ] 구현체 이름이 의존 대상을 드러내는가 (`Impl`/`Default` 없음)
- [ ] 새 인터페이스가 §2 허용 경계 표 안에 있는가
- [ ] 사용처 없는 코드·파라미터·주석 코드가 없는가
- [ ] 계층 패키지(port/adapter/usecase 류)·중복 DTO 계층이 없는가
- [ ] 패키지가 하나의 관심사로 설명되는가 (10파일 초과 시 분리 검토)
- [ ] 생성자 주입·불변·예외 비은폐를 지켰는가
- [ ] (공통) spec 범위 안의 변경인가, `.env`·`data/`·`logs/`가 새어 들어가지 않았는가
